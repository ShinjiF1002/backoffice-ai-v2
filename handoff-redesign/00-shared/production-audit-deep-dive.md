# 本番監査 Deep-Dive — 銀行が最重視する 4 次元 (audit 補強)

> `production-readiness-audit.md` の §3 多次元 gap を、銀行 IT監査・リスク・コンプラ・AI ガバナンスが**特に深く問う**領域で掘り下げた補強。最終的に audit へ merge する。
> 注: 規制は `CLAUDE.md` の hedge 方針に従い、特定規制への事実主張をせず**要件カテゴリ**として扱う ([将来確認項目])。

## DD-1. AI ガバナンス・モデルリスク (銀行 model risk management の核心)

中核業務が「AI が値生成 → 人が確認 → 段階自動化 (Trust 昇格)」である以上、銀行の **model risk management** 観点が最重要。現状 prototype に欠落:

| ID | gap | なぜ銀行が問題視 | Sev |
|----|-----|----------------|-----|
| DD-1.1 | **モデル版管理の証跡なし** | どのモデル版が各案件を処理したか追跡不能。モデル更新時の影響範囲・再現性が担保できない (監査で「この判断はどのモデル?」に答えられない) | P0 |
| DD-1.2 | **「AI が自信を持って間違える」検出の運用が UI のみ** | reconcile-panel-spec の検出軸 (独立ソース突合/業務ルール/異常値/前回差分/同時更新/checker sampling) は設計にあるが、実 pipeline・監視・閾値運用が無い。高 confidence の誤りが自動確定し得る | P0 |
| DD-1.3 | **confidence 抑制と監査の緊張が未解決** | UI は confidence 非表示 (正)、台帳のみ可。しかし**運用監視 (drift/誤り率)・モデル評価には confidence の集計が必須**。集計・監視 dashboard が無い | P1 |
| DD-1.4 | **drift / bias 監視なし** | 入力分布の変化・特定法人類型での誤り偏りを検知できない。承認率 KPI は事後・粗い | P1 |
| DD-1.5 | **Trust 昇格/降格の統制が gate 表示のみ** | 昇格基準 (4 KPI) は表示するが、昇格の**承認権限・記録・自動降格の実装・champion-challenger / shadow-mode 検証**が無い。「要所確認」昇格後に品質劣化したときの安全網が設計のみ | P0 |
| DD-1.6 | **human-override の証跡が業務 event のみ** | AI 入力を人が上書きした事実は記録されるが、**override 率・override 理由の分類・モデル改善への feedback loop** がモデルガバナンスとして閉じていない | P1 |
| DD-1.7 | **説明可能性 (explainability)** | なぜ AI がその値を生成したかの根拠 (source locator はあるが reasoning なし)。規制対応・異議申立対応で不足 | P1 |
| DD-1.8 | **third-party / 基盤モデルのベンダーリスク** | 外部 LLM/OCR 利用時のデータ越境・可用性・版固定・契約上の説明責任 | P1 |

→ **示唆**: 是正計画 R3-2 (AI モデルガバナンス) を**最重量級**に格上げ。model registry / monitoring / override-analytics / Trust-governance を独立 sub-system として設計。

## DD-2. 規制・コンプライアンス要件 (要件カテゴリ、[将来確認項目])

| ID | gap | なぜ銀行が問題視 | Sev |
|----|-----|----------------|-----|
| DD-2.1 | **訂正・取消 (reversal/correction) flow が無い** | 銀行は誤って反映した登録情報を**正式手順で訂正・取消**できねばならない。現状 status 遷移は前進のみ。誤反映後の是正経路・その証跡が設計に無い | P0 |
| DD-2.2 | **maker-checker の完全性** | 入力者→承認者は表現されるが、**手順承認・設定承認の maker-checker、提案却下後の再申請、承認の差戻し連鎖**の完全性が未検証。承認の各段で「誰が・いつ・何を根拠に」が法的に閉じるか | P1 |
| DD-2.3 | **SoD のシステム強制 (再掲・深掘り)** | mode toggle で自己切替可能 = SoD は表示のみ。**同一人が入力者かつ承認者になれない**ことをシステムが拒否する実装が必須。代理・委譲・不在時のエスカレーション規程も | P0 |
| DD-2.4 | **保持・廃棄・WORM** | 監査証跡・案件データの**保持期間・改竄防止 (WORM/hash chain)・期限後廃棄**ポリシー。e-discovery 出力 | P0 |
| DD-2.5 | **変更管理の法的有効性** | Agent 設定変更・ルール改定の承認が**正式な変更管理記録**として成立するか (承認者の本人性・タイムスタンプの信頼性・否認防止) | P1 |
| DD-2.6 | **個人情報・機密区分** | 法人/個人情報の取扱区分、画面表示の最小権限、マスキング、目的外利用防止 | P1 |
| DD-2.7 | **規制報告・traceability** | 規制要件→画面/操作/証跡の traceability matrix。監査対応時に「要件 X はどこで満たすか」を即答できる構造 | P1 |

→ **示唆**: 是正計画に **R3-5「訂正・取消 flow」を新設** (P0、現計画に欠落)。reversal は銀行で必須だが prototype に完全欠落。

## DD-3. 運用レジリエンス・BCP

| ID | gap | なぜ銀行が問題視 | Sev |
|----|-----|----------------|-----|
| DD-3.1 | **AI 不能時の手動 fallback なし** | OCR/LLM 障害時に業務が止まる。**AI なしで人が全項目手入力して処理を継続**できる degraded mode が必須 | P0 |
| DD-3.2 | **冪等性・二重処理防止** | 再送・二重 click・retry で同一案件が二重反映されない保証。現状 button 連打可 (X-10/D9) | P0 |
| DD-3.3 | **再処理 (差戻し→AI 再入力) の安全性** | 差戻し後の再処理で前回データとの整合・無限ループ防止・version 管理 | P1 |
| DD-3.4 | **flywheel の失敗モード** | staging への悪質/誤ヒント混入 (staging poisoning)、誤った提案の自動適用、日次分析の品質劣化の検知・遮断 | P1 |
| DD-3.5 | **failover / 可用性 SLA** | 業務時間中の可用性、計画/非計画停止時の継続性、データ整合の回復 | P1 |
| DD-3.6 | **incident response** | AI 誤判断が大量発生したときの一括停止 (kill switch)・影響案件の特定・遡及訂正 | P0 |

→ **示唆**: R4-3 (レジリエンス) に **kill switch + AI-fallback + 冪等性** を P0 として明記。現計画では P1 扱いだったが格上げ。

## DD-4. IA / UX の構造的弱点 (BO power-user at scale)

| ID | gap | なぜ問題か | Sev |
|----|-----|----------|-----|
| DD-4.1 | **一括操作 (bulk) なし** | 数百件を 1 件ずつ処理は非現実的。複数選択→一括承認/差戻し/担当割当が必須 (SoD 範囲内で) | P1 |
| DD-4.2 | **キュー所有・割当なし** | 「誰がどの案件を担当」の割当・ロック (他者が同時処理しない)・引き継ぎ。現状 担当列は表示のみ | P1 |
| DD-4.3 | **cross-process / 例外案件の扱い** | Process-First IA は業務単位。複数業務にまたがる案件・どの業務にも属さない例外の動線が無い | P2 |
| DD-4.4 | **保存ビュー・個人設定なし** | role landing / saved view は spec にあるが未実装。各人の作業効率に直結 | P2 |
| DD-4.5 | **3 層承認の動線の穴** | 案件承認→手順承認→設定承認の遷移・差戻し連鎖・各層の権限境界が画面横断で閉じているか未検証 (E2E で全行同一 detail のため未確認) | P1 |
| DD-4.6 | **通知・SLA エスカレーション** | 期限超過の能動通知、担当への push、未処理の上長エスカレーション。Hub の SLA カードは表示のみ | P1 |

→ **示唆**: R0 に **bulk 操作 + キュー割当/ロック** を追加 (層 A で client-side でも価値)。R1 で実 lock。

## 是正計画への反映 (この deep-dive が追加する項目)

1. **R3-2 AI モデルガバナンスを最重量級に** (model registry / drift・bias 監視 / override analytics / Trust governance / shadow-mode)。
2. **R3-5 訂正・取消 flow を新設** (P0、現計画欠落)。
3. **R4-3 に kill switch + AI fallback + 冪等性を P0 明記**。
4. **R0 / R1 に bulk 操作 + キュー割当・ロックを追加**。
5. **DD-1.3 confidence 監視 dashboard** (UI 非表示と運用監視の両立) を R3 に追加。

→ Codex round 1 findings と reconcile 後、`production-remediation-plan.md` を v1.0 に更新する。
