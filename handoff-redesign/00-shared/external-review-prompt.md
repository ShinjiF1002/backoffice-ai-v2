> **使い方**: 以下を別 AI (ChatGPT / Gemini / Claude.ai 等) に paste。可能なら添付 2 文書 (`operational-audit.md` / `ia-overview-v2.md`) + 元 mockup HTML 6 枚も一緒に upload。添付がなくても本 prompt 内の要約だけでレビュー可能なよう self-contained に構成済。

## あなたへの依頼

あなたは **エンタープライズ SaaS の UX 設計 + 業務オペレーション + 規制対応** に精通したレビュアーです。以下に示す「銀行バックオフィス業務向け AI エージェント prototype の UI/UX 再設計方針」を **critical に** review してください。

**重要**: 私 (依頼者) はこの方針を自分で策定したため confirmation bias があります。**鵜呑みにせず、欠陥・見落とし・過剰設計・実務との乖離を積極的に指摘**してください。「良い」だけのレビューは不要です。

## 1. プロジェクト背景

- **対象**: 大手銀行のバックオフィス (BO) 業務に AI エージェントを段階的に導入する構想の UI prototype
- **目的**: 経営層向けの構想共有 demo (実装ではなく設計の妥当性検証フェーズ)
- **中核思想**: 「差戻しを、次の正解手順に変える仕組み」
  - AI が業務を処理 → 人間が確認/差戻し → 差戻しを暫定知識として記録 → AI が日次分析して手順改定を提案 → 承認されたら正式手順に昇格
  - スローガン: 「案件確認は減らす。ルール承認は残す」(AI に任せる量は段階的に増やすが、人によるコントロールは渡さない)
- **3 層承認**:
  1. **案件承認** = 個別案件の AI 入力結果を人が確認。**4-eyes 構造** (入力者確認 + 承認者承認、SoD: 入力者 ≠ 承認者)
  2. **手順承認** = 差戻しコメントを正式手順に昇格 (AI が提案を自動生成 → Manual 管理者が triage → 業務責任者が承認)
  3. **設定承認** = AI エージェントの権限/モデル/プロンプト変更を承認
- **Automation Maturity** (信頼度段階): Supervised (全件 4-eyes) → Checkpoint (重要分岐のみ) → Autonomous (サンプリング、将来)
- **業務 Process は 2 つ**: 「法人住所変更」「口座開設書類完備チェック」。エージェントは各 Process に 1:1 対応
- **role は 6 つ**: 入力者 / 承認者 / Manual 管理者 (提案キュー責任) / 業務責任者 (最終承認) / AI 管理者 (エージェント設定) / 監査者
- **制約**: Day1 (運用初期) scope / 日本語 UI のみ / 規制・統制配慮 / 実 LLM 呼び出しや外部システム接続はなし (すべて mock データ)

## 2. 再設計に至った経緯

1. 元の UI (9 画面) に 3 つの認知負荷問題があった: ①文字が多すぎる ②一目で何をすべきか分からない ③画面ごとにレイアウトが不統一
2. 一次再設計で機能を 9→6 画面に集約し、デザインツールで高品質モックを 12 枚作成した
3. しかし利用者レビューで **16 の構造的指摘** が出た (下記 §3)。これらは「機能集約」では解けず、より根本的な再設計が必要と判明
4. そこで、6 つの role の実オペレーション walkthrough (operational journey audit) を起こし、各 step で必要な情報/操作/検証を洗い出して **18 要件** を逆算 (添付 `operational-audit.md`)
5. その 18 要件を満たす **Process-First IA (9 画面 + 横断 component 4 種)** を策定 (添付 `ia-overview-v2.md`)

**今回レビューしてほしいのは、4-5 の「再設計方針」が妥当か** です。

## 3. 利用者レビューで出た 16 指摘 (元の問題意識)

### 監査・メトリクス・ナレッジ画面 (Observatory)
1. 監査証跡は業務 Process ごとであるべき (現在は特定 Process のみ)
2. AI による入力がいつ行われたか監査証跡に現れていない
3. 時系列が論理矛盾 (人による入力者確認の後に AI が OCR 処理している。確認の後は承認者の承認であるべき。また「cron trigger」という内部用語が利用者に意味不明)
4. メトリクス画面は業務 Process ごとであるべき (各 Process の AI 精度や KPI を見たいはず)
5. ナレッジ一覧も業務 Process ごとに表示すべき (利用者は Process 起点で考える)

### エージェント設定画面 (AgentDetail)
6. 信頼度昇格申請で利用者が気にするのは「各指標の実数値と、事前定義した閾値との比較」。現在の「進化要件達成度 75%」のような集約値ではなく、指標ごとのパフォーマンスを表示すべき
7. 特定エージェント画面の前に、エージェント一覧画面があるべき
8. 信頼度変更で何が起こるか (Consequence) が画面で分からない。「全件人レビューだったものがどう変わるか」を記載すべき

### 提案レビュー画面 (ProposalDetail)
9. 提案レビューの前に「どの Process のどの部分の話か」が分かる一覧画面が必要
10. 判定基準は良いが、実績値がいくらなのか分からない
11. このルール改定で何がどう変わるかが画面で分からない

### 案件詳細画面 (CaseDetail)
12. AI 入力結果に 0.96 や 0.84 の数字があるが意味不明。何を測定し何の意図で表示するか不明 → 表示すべきでない
13. 「差戻し (コメント付き)」ボタン押下時、コメント未入力なら即エラーで入力を促す UX であるべき
14. **AI 入力結果と、申請書類 (OCR 等) の内容を突合 (reconcile) する UI であるべき**。利用者がこの画面で業務を完結するために必須。本件に限らず全画面で「実際の利用者オペレーションを想定した設計」になっていないため、まず網羅的監査と証跡ドキュメントを作るべき
15. 「0.84 / field 平均 (5 field)」は意味不明 → 削除

### 案件一覧画面 (Queue)
16. 右パネルはデフォルト非表示にし、明細クリックで表示すべき / パネル内の 0.96, 0.84 は意味不明で削除 / パネルで「何が変わったか」が分かるべき

### 全社状況画面 (Hub)
17. 「注意あり」「SLA 超過」などの KPI が、どの Process の Alert か分からない。利用者は「特定 Process のエージェント設定の問題か、低品質リクエストの大量発生か」など原因を考えながら操作するはず

## 4. 私が策定した対応方針 (これをレビューしてほしい)

16 指摘の背後に **5 つの構造テーマ** を抽出した:
- **T1 業務 Process 軸の欠落** (最頻出): UI が機能横断のフラット集約で、利用者の「Process 起点」メンタルモデルに不一致
- **T2 master-detail の master 欠如**: エージェント一覧・提案一覧・承認者キューが存在しない
- **T3 意思決定支援情報の欠如**: 実数値 vs 閾値 / 変更影響 (before-after) / 突合結果がない
- **T4 confidence 生数字の露出**: 0.84 等を行動に紐づけず表示
- **T5 実オペレーション完結性の欠如**: 突合 (reconcile) / 入力検証 / 監査 timeline の論理矛盾 / 承認者画面の不在

### 4.1 18 要件 (operational journey audit から逆算、全 Day1 実装可)

|---|---|---|
| R-PROC-01 | 業務 Process を第一階層の組織軸にする (Sidebar = Process scoped) | P0 |
| R-QUEUE-01 | 案件一覧の右パネルは default 非表示、行クリックで開く | P0 |
| R-QUEUE-02 | 右パネルで「何が変わったか」(変更内容) を表示 | P1 |

### 4.2 Process-First IA (現 6 → 新 9 画面)

**nav 構造**: TopBar に「業務 Process Selector」([法人住所変更 ▾ / 口座開設 / 全業務])。Sidebar の機能 nav は選択中 Process に scoped (Linear の team switcher / Stripe の account switcher pattern)。

**9 画面** (typology: A=Hub型 / B=一覧 master / C=詳細):

| # | 画面 | typology | 新設? | 主目的 |
|---|---|---|---|---|
| 1 | Hub | A | 改修 | 全 Process の Alert を Process tag 付きで集約、原因究明 drill |
| 2 | 案件一覧 (入力者) | B | 改修 | 担当案件の queue |
| 3 | 承認待ち (承認者) | B | ★新設 | 4-eyes 後半 |
| 4 | 案件詳細 | C | 改修 | 突合 UI + 検証 + 入力者/承認者 mode |
| 5 | 提案一覧 | B | ★新設 | Process 別の手順改定提案 |
| 6 | 提案詳細 | C | 改修 | 実測値 + before/after |
| 7 | エージェント一覧 | B | ★新設 | 全エージェントの信頼度/パフォーマンス |
| 8 | エージェント詳細 | C | 改修 | 実指標 vs 閾値 + Consequence |
| 9 | Observatory | A | 改修 | Process 別の監査/メトリクス/ナレッジ (3 tab) |

### 4.3 横断 component 4 種 (新規、複数画面で共用)

1. **ProcessSelector** (TopBar): 業務 Process の切替。全 nav に Process context を供給
2. **ReconcilePanel** (案件詳細 / 承認待ち詳細 / 案件一覧の右パネル): confidence 生数字を捨て、「AI 入力値 vs 申請書類 (OCR) vs マスタ の 3 者突合」を表示。一致 (緑) / 要確認 (黄) / 未取得 (灰) の 3 状態。利用者は「人が確認すべき項目」だけ見ればよい。ルールベース (文字列一致/差分) で実装し ML 不要
3. **MetricVsThreshold** (エージェント詳細 / 提案詳細): 「達成度 75%」のような集約値を捨て、各指標の「実績値 vs 閾値 vs 達成判定」を表示
4. **ConsequencePanel** (エージェント詳細 / 提案詳細): 信頼度昇格やルール改定の before/after・影響件数を表示

### 4.4 confidence の扱い (Day1 / 将来の切り分け)

- 現状の 0.84 等は「OCR エンジンの文字認識確信度」想定だが、利用者には解釈不能と判断
- **Day1**: 生数字を全削除し、上記 ReconcilePanel の「一致/要確認」に置換 (ルールベース、ML 不要)
- **将来**: 過去事例で学習した ML calibrated confidence は学習データ蓄積後に検討。表示方法も項目毎の数字ではなく別の UX (優先確認順 等) を別途設計

## 5. レビューしてほしい観点 (critical に)

各観点について **強み / 弱み / 見落とし / 推奨修正** を述べてください。

1. **IA の妥当性**: Process-First (Global Selector + 9 画面) は、業務 Process が 2 つしかない現状で過剰ではないか? 逆に、Process が将来 10+ に増えた時に破綻しないか? 9 画面への増加は認知負荷を本当に下げるか (画面が増えれば迷いも増えるのでは)?
2. **実オペレーション完結性**: 6 role の journey は現実的か。見落とした role / step / 例外フロー (エスカレーション、代理承認、SLA 超過時の挙動など) はないか?
3. **横断 component の設計**: 特に **ReconcilePanel** — 「AI 入力 vs 申請書類の突合」を Day1 にルールベースで実現する設計は妥当か? 文字列一致では拾えない不一致 (表記ゆれ、意味的同一) をどう扱うべきか? **MetricVsThreshold / ConsequencePanel** の情報粒度は意思決定に十分か?
4. **confidence → reconcile 置換**: 生 confidence を捨てて突合に置き換える判断は正しいか? 突合だけでは拾えない「AI が自信を持って間違える」ケースをどう検出するか?
5. **規制・統制の観点**: 4-eyes (入力者≠承認者) / 監査証跡の完全性 / 操作の否認防止 / 権限分離 の観点で、UI に欠落や危険はないか? 監査 timeline の設計 (ライフサイクル順) は監査要件を満たすか?
6. **16 指摘の充足**: §3 の 16 指摘が §4 の方針で本当にすべて解消されるか? 表面的に対応しているが本質的に解けていない指摘はないか?
7. **見落とし**: 16 指摘以外に、実際の銀行 BO 利用者がこの UI を使った時に困る点はないか?
8. **最大リスク**: この再設計方針の最大のリスクは何か? (実装難易度、利用者の学習コスト、demo での説明しにくさ、scope 肥大 など)

## 6. 出力形式の指定

以下の構造で返してください:

1. **総合評価** (3-5 文): この方針で実装に進んでよいか。条件付きなら条件を明示
2. **強み** (箇条書き、最大 5): 特に良い設計判断
3. **弱み・欠陥** (表形式): `観点 / 指摘 / severity (P0=実装前に必須修正 / P1=実装中に対応 / P2=将来) / 推奨修正`
4. **見落とし** (箇条書き): 19 指摘 (F-01〜F-19) / 22 要件 (R-*) でカバーされていない点 (正確な対応は添付 `coverage-matrix-v2.md`)
5. **横断 component への具体 feedback**: ReconcilePanel / MetricVsThreshold / ConsequencePanel / ProcessSelector の改善案
6. **最大リスク 3 点** と緩和策
7. **coverage matrix の充足確認**: 19 指摘 (F-01〜F-19) × 22 要件 (R-*) × 9 画面 × component の対応に漏れがないか。どの画面でも閉じない指摘があれば指摘してください
8. **現物 drift の監査**: 方針 doc (`ia-overview-v2` / `screen-contracts-v2`) と handoff 実体 (prompt / mockup) に乖離がないか。特に **旧 6 画面の `confidence` / `cron` / `75%` が v2 側に混入していないか**

忖度なしで、実務に耐えるかを厳しく評価してください。
