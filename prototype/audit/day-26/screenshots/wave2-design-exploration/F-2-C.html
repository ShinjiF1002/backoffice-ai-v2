設計意図

Card 2 ★ の 3 view 全部 を強制揃えで露出し、CaseReview / ProposalReview の 両 page で同一 API + 同一 component として共有する統一案。 Field table を default にすることで「5 field の audit grid を最初に目にする」体験を作り、Inline / Side-by-side は局所確認用に切替可能。 MetadataStrip を PageHelpDisclosure 連携の collapsible にすることで、density を下げつつ詳細は on-demand に展開できる (5 element 強制露出を緩和)。

採用適性

最も card 完全準拠。ただし両 page を 1 component で書く強い制約があり、ProposalReview の手順 markdown 差分には Field table がうまく当てはまらない (項目構造ではないため)。 Side-by-side のみで運用する場面が多くなり、tab toggle が 常時露出しすぎる リスクがある。
