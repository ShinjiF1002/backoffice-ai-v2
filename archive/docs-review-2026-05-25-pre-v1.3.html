DOC-OV-00

構想概要 (Overview)

1. 課題と中核 message

1.1 解く課題

バックオフィス業務は、ベテラン担当者の暗黙知に依存してきた領域である。組織として AI 導入を検討するとき、選択肢は実務上「一括自動化 (高リスク、例外対応の難しさ・規制対応の重さで実装が止まる)」か「諦める (現状維持、人手不足と退職リスクで持続不可能)」の二択に陥りやすい。

v2 が解こうとするのは、この二択の間に 段階的な自動化 + 暗黙知の組織知への蒸留 という第 3 の選択肢を提示することである。

1.2 中核 message

業務別ナレッジ文書を人間承認で育てる Flywheel

  • AI が業務手順を実行 → 入力者が結果を確認 / 差戻し
  • 差戻しコメントは即時 staging knowledge として AI runtime に visible になる
  • 複数 case で再発した staging は batch で手順承認を経て compiled knowledge に昇格、業務手順ファイル群 (workflow.md / agent-instructions.md / approval-policy.md) に diff 適用
  • 手順変更が AI 設定 (Agent / Model / Tool / Prompt / 権限) に及ぶ場合は設定承認を通る
  • Automation Maturity (Supervised / Checkpoint / Autonomous) は知識・設定承認 loop を縮小せず、業務承認 (案件承認) の介在頻度だけ縮小する

「コメントを貯める仕組み」ではなく「業務別ファイルを育てる仕組み」として設計するのが v2 の差別化点。

2.2 起点 (本番想定の flow) と v2 prototype の見せ方

本番想定の flow
  1. 依頼書 PDF が特定フォルダに置かれる
  2. AI watcher が検知、自動的に業務システム操作を開始
  3. 入力結果 (スクショ stack + PDF preview + Alert) を入力者の Case Review 画面に提示
  4. 入力者確認: 承認 / 差戻し
  5. 入力者承認後、承認者承認: 同じ証跡を見て最終確認

入力者確認 + 承認者承認 が揃って 案件承認 全体を構成。「4-eyes」はこの 2 者が揃った案件承認全体に対する呼称、単独 / 別々には使わない。

v2 prototype の見せ方 (非実行)

  • Case Review 画面 (/cases/:id) で mock evidence (PDF サムネ / スクショ stack / Alert / agent proposal) を表示
  • 入力者確認 / 承認者承認 の遷移は mock state 更新のみ、業務システム実操作 / 永続化 は scope-out

2.3 UI scope

  • in-memory mock state、永続化なし

  • 承認者画面は実装せず、CaseReview 内 BusinessApprovalChip + slide-only static mock で代替

2. ① 起点: 入力者確認による差戻し

2.1 入力者の役割

入力者は 4-eyes flow の 1 段目で AI 入力結果を確認する。これを 入力者確認 と呼ぶ。

入力者確認 単独では 4-eyes と呼ばない。4-eyes は 入力者 + 承認者 の 2 者が揃って案件承認全体を構成するときの呼称。

2.2 差戻しの構造 — 5-category

入力者は SendBackComment 画面 (/cases/:id/comment) で 5-category + free-text コメント + case context (PDF / screenshot / agent output) を送信。

  • prototype: 同一セッション内に mock state を AppContext で immediate update、永続化なし

4.1 昇格 trigger

  • Manual 管理者が curate して proposals/pending/ に手順更新提案を起票

4.3 反映範囲 (diff 適用)

  • workflows/{業務}/workflow.md
  • workflows/{業務}/agent-instructions.md
  • workflows/{業務}/approval-policy.md
  • workflows/{業務}/knowledge/staging/*.md → .../compiled/*.md (move、weight 上げ)

workflows/{業務}/BOUNDARY.md は通常 loop の更新対象外。BOUNDARY.md 変更は scope 拡張・自動化禁止条件見直しを伴うため、明示的な設定承認 + plan update を経る別 process で扱う。Flywheel の通常 loop で staging → compiled → BOUNDARY.md の diff という経路は発生しない。

5. ④ 設定承認 (条件付き)

5.1 trigger 条件

手順承認の影響範囲が AI 設定 (Agent / Model / Tool / Prompt / 権限) に及ぶ場合のみ trigger:

  • agent-instructions.md の更新で新 tool 呼び出しが必要
  • 参照ナレッジ範囲拡大で Agent の権限変更が必要
  • Automation Maturity の昇格 (Supervised → Checkpoint)

Type A / B / C の詳細 は DOC-APP-02 §4 設定承認 が SSOT。本 doc は Flywheel における trigger 経路 + §5.3 手順承認のみで完結する case の切り分けのみ扱う。

6. ⑤ 反映後の Case Review への波及

6.1 新 case の citation

  • compiled knowledge を AiProposalPanel の citation list に明示
  • confidence score 上昇 (引用ナレッジ ↑ → 不確実性 ↓)
  • Alert ルール更新 (approval-policy.md 反映時、新 case から適用)

6.2 KPI 観測

  • AI 入力承認率 ↑ (差戻し率 ↓)
  • 人手上書き率 ↓
  • Alert 命中率 (新 rule の有効性)

数値は全件 [仮説] ラベル

6.3 過去 case は遡って書き換えない

反映後も過去 case の AI proposal は不変。理由: audit trail 保護 (過去判断の根拠を後から書き換えない) / 入力者・承認者の判断履歴を変更しない / 「いつ何が変わったか」を Audit Trail (/audit) で trace できる。Mock data でも踏襲。

7. Loop の不変条件 ★ Session 4 Slide 7 message の核

7.1 Matrix B 原則 (Automation Maturity との関係)

Automation Maturity が進化しても、知識・設定承認 loop は縮小しない。縮小するのは案件承認 (= 入力者確認 + 承認者承認) の介在頻度のみ。

Automation 段階 案件承認の介在 手順承認 設定承認
Supervised 全件 (入力者 + 承認者) 通常 通常
Checkpoint 重要分岐のみ 通常 通常
Autonomous (将来) サンプリング 通常 通常
7.2 出典 (prior art pointer)

ai-operator/docs/01-hitl-policy.md line 198-204 (Matrix B 該当)。外部規制根拠ではなく、ai-operator paper の設計判断を v2 が継承したもの。本 v2 では prior art reference として扱い、外部規制論拠 (NYDFS / SR 11-7 等) としては主張しない。

7.3 Loop は単方向 / 7.4 停止条件

  • 差戻し → staging → compiled → 手順承認 → (設定承認) → 反映 の順に進む
  • staging から workflow.md に 直接 diff は飛ばない (compiled 昇格 + 業務責任者承認 必須)
  • BOUNDARY.md は通常 loop の更新対象外 (§4.3)
  • v2 scope 内では「差戻し 0 件」「threshold 未達」「業務責任者 reject」の 3 case のみ Loop 停止
  • 緊急停止 (Forced Downgrade / Emergency stop) は DOC-APP-02 Matrix C の範疇、本 doc scope 外

2. 案件承認 (Case Approval)

2.2 4-eyes 構造

段階 役割 内容 反映タイミング
入力者確認 入力者 AI 入力結果を accept / 差戻し (5-category + free-text) 送信時 (prototype では同一セッション内)
承認者承認 承認者 入力者承認済の最終確認 承認時 (prototype では同一セッション内)

4-eyes はこの 2 者が揃った案件承認全体を指す呼称。入力者確認単独 / 承認者承認単独に対しては使わない。SoD: 入力者 ≠ 承認者 (システム制約として強制、同一人物による self-approval は禁止)。

2.4 エスカレーション条件

  • Alert 発火 + 入力者・承認者の判断不能
  • 業務責任者の介入が必要 (高リスク要素検知)
  • データ整合性異常 (5-category の data_error)

3. 手順承認 (Procedure Approval)

差戻しコメントを正式手順に反映する承認。knowledge → procedure / workflow.md / agent-instructions.md / approval-policy.md への昇格 trigger。

入力者 / 承認者

SoD: Manual 管理者 ≠ 業務責任者 (システム制約として強制、起票者と承認者の同一人物化は禁止)。

4. 設定承認 (Configuration Approval) — Type A/B/C

Type A 通常 co-A なし
対象: Prompt minor update / 既存 Tool 範囲内 / Model patch release
R: AI 管理者 / A: AI 管理者 (起票者 ≠ 承認者の SoD 強制)
例: Agent-X の prompt v1.3 → v1.4
Type B Security-impacting co-A 必須
対象: 新 external tool / 権限拡張 / data boundary 変更 / 認証方式変更
R: AI 管理者 / A: AI 管理者 + Security 関係者
例: 外部 vendor LLM 新規採用、PII scope 拡大
Type C Automation Maturity 変更 co-A 必須
対象: Supervised → Checkpoint → Autonomous の段階変更
R: AI 管理者 / A: AI 管理者 + 業務責任者 (業務リスク受容)
例: UC-BO-01 法人住所変更処理を Supervised → Checkpoint
4.3 Type A の SoD (重要)

Type A は AI 管理者の承認 path だが、同一 change の起票者本人によるセルフ承認は禁止。起票者 ≠ 承認者であることをシステム制約として強制。

  • 複数 AI 管理者体制 (default): 組織内の AI 管理者ロールを持つ別人物が承認
  • 1 人体制の組織 (fallback): Type A も co-A (= 業務責任者 または Security 関係者) を必須とする
  • 起票と承認を別 commit / 別 timestamp で記録、AuditTrail に SoD evidence として残す
